# Contributing

Thank you for contributing to **tigr-asgi-contracts**.

This repository is the neutral source of truth for the shared ASGI3-compatible contract consumed by **Tigrcorn** and **Tigrbl**. The contract is published in two forms:

- **artifact packages**: YAML registries, JSON Schemas, manifest, checksums
- **downstream contract packages**: generated Python, Rust, and TS / TSX contracts

## Principles

1. `contract/` is the only canonical source-of-truth directory.
2. Generated outputs are derived products and must never be edited by hand.
3. `security_deps` semantics, exchange semantics, family semantics, and subevent semantics are owned by the contract, not downstream implementations.
4. Pull requests must leave the repository in a regeneration-clean state.

## Authoring workflow

The exact authoring sequence is:

1. Edit files under `contract/`.
2. Run validation.
3. Rebuild `manifest.json` and `checksums.txt`.
4. Generate downstream contract packages.
5. Run tests.
6. Commit **both** the source contract changes and the regenerated downstream outputs.

### Local commands

```bash
python tools/validate_yaml.py
python tools/validate_jsonschema.py
python tools/build_manifest.py
python tools/build_checksums.py
python tools/generate_all.py
python tools/check_versions.py
pytest -q
```

## Do we commit generated outputs?

**Yes.**

Generated outputs under:

- `packages/contract-py/`
- `packages/contract-rs/`
- `packages/contract-npm/`

are committed to the repository.

They are reviewed, diffed, and versioned alongside the canonical contract.

## What GitHub Actions generation is for

GitHub Actions also runs generation, but **only to verify determinism**.

CI generation:
- must not push commits
- must not become the source of truth
- must fail if regeneration changes tracked files

That means the correct policy is:

- **authors generate and commit locally**
- **CI regenerates and verifies clean state**

## Pull request checklist

- [ ] only `contract/` was edited by hand
- [ ] `python tools/generate_all.py` was run
- [ ] `pytest -q` passes locally
- [ ] no uncommitted generated changes remain
- [ ] `VERSION` and package versions remain aligned
- [ ] documentation was updated if the public contract changed

## Release sequence

The release sequence is intentionally strict:

1. **Generate**
2. **Test**
3. **Promote** (create immutable release-candidate bundle)
4. **Publish** (publish the promoted bundle to registries)
5. **Release** (finalize the public GitHub release after publish succeeds)

### Why publish before final release?

Because a public GitHub release should only exist once ecosystem publication succeeded. A failed registry publish should not leave behind a misleading final release.

## Promotion policy

Promotion creates a release-candidate bundle that is:

- version-checked
- regeneration-checked
- test-checked
- package-built
- uploaded as immutable workflow artifacts
- attached to a GitHub prerelease

That promoted candidate is the boundary between CI validation and external publication.

## Publishing policy

- PyPI and npm publish from the promoted bundle assets.
- crates.io publishes from the promoted source tag / commit.
- No workflow should mutate tracked generated sources during publish.

## Project areas

| Area | Hand-editable? | Notes |
|---|---:|---|
| `contract/` | Yes | source of truth |
| `generators/` | Yes | code generators and templates |
| `packages/artifacts-*` | Yes | package plumbing only |
| `packages/contract-*` | No, except generated by tooling | generated outputs |
| `tests/` | Yes | validation, parity, and packaging tests |
| `.github/workflows/` | Yes | release and verification automation |

## Development environments

Recommended tools:

- Python 3.11+
- Node 20+
- Rust stable
- `pytest`
- `PyYAML`
- `jsonschema`
- `build`

## Reporting issues

Please include:

- contract version
- exact file(s) changed under `contract/`
- generated diff summary
- failing command / workflow step
- expected vs actual behavior
